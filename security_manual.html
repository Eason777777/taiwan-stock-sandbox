Security Manual

Foundry 安全審查工具包 — 不需要 API key，搭配 Claude Code 使用

✅
本手冊說明的功能只需要 Claude Code CLI（Claude Pro 即可）， 不需要 Anthropic API key。

概覽

Foundry 是一個 AI 輔助的程式碼安全審查框架。它有兩個使用層次：

層次檔案需要 API key？用途
偵測規則foundry/rules/corpus/*.yaml否Claude Code 的掃描 checklist
程式碼解析器foundry/index/parser.py否純 Python，tree-sitter 解析
單元測試tests/unit/否驗證核心邏輯
完整自動化代理人foundry/roles/是無人值守自動掃描

本手冊的使用方式：你把程式碼交給 Claude Code， Claude Code 對照偵測規則進行分析，找出安全漏洞。

快速開始

1

安裝依賴（選用）

僅在需要用程式碼解析器時才需要安裝：

conda activate sqlproject
pip install tree-sitter tree-sitter-python pyyaml pytest
2

驗證環境

pytest tests/unit/ -v
# 預期輸出：34 passed
3

開啟 Claude Code，請它進行審查

# 在 repo 根目錄開啟 Claude Code
claude

# 或在 IDE 的 Claude Code 插件中輸入：
請對照 foundry/rules/corpus/ 裡的偵測規則，
掃描 app/ 目錄下的安全問題。

規則 YAML 結構

id: SQL-001                     # 唯一識別碼
name: SQL Injection              # 漏洞名稱
vulnerability_class: SQL Injection
cwe: CWE-89                     # CWE 分類
severity_hint: high             # critical / high / medium / low
applicable_languages: [python]  # 適用語言
scope: function                 # 分析粒度
prompt_template: |              # ← 這是最重要的欄位
  分析以下函式是否有 SQL injection。
  需要同時滿足：
  (1) SQL 送到資料庫執行
  (2) 輸入來自函式外部（HTTP、env、檔案）
  (3) 輸入沒有經過 parameterization 或 sanitization
  ...
  {{function_body}}

prompt_template 精確描述了「什麼情況才算真正的漏洞」。 直接閱讀這個欄位就能理解判斷標準。

三段式證據標準

Foundry 要求每個確認的漏洞必須有三段獨立論證。缺任何一段就不算是真正的漏洞。 請 Claude Code 做分析時也應要求它遵循這個標準。

三段式證據閘門（Evidence Gate）
1
Reachability — 可達性
攻擊者如何控制這個輸入？ 從哪個 HTTP 參數、環境變數、或外部資料來源進來的？ 有沒有任何過濾擋在中間？
2
Trust Boundary — 信任邊界
不受信任的資料在哪個具體的位置（檔案、函式、行號） 進入了受信任的處理流程？有沒有驗證或 parameterization 在這個邊界上？
3
Impact — 安全影響
攻擊者可以造成什麼具體的後果？ 資料外洩？任意命令執行？帳號接管？ 需要是確定的影響，不是「可能有影響」。

審查指令範本

全面掃描

請閱讀 foundry/rules/corpus/ 下的所有規則，
然後對 app/ 目錄進行全面的安全審查。

對每個發現的問題，請說明：
1. 位置（檔案名稱和函式名稱）
2. 漏洞類型（對應哪個規則 ID）
3. Reachability：攻擊者如何控制輸入
4. Trust Boundary：在哪裡跨越了信任邊界
5. Impact：具體的安全後果
6. 修復建議

針對特定漏洞類型

請參考 foundry/rules/corpus/sql-injection.yaml 的判斷條件，
掃描 app/routers/ 下所有和資料庫互動的函式。

只回報同時滿足三個條件的漏洞：
(1) SQL 字串包含外部輸入
(2) 沒有使用 parameterization
(3) 這個函式可以從外部（HTTP/使用者輸入）被呼叫

針對特定檔案的深度分析

請仔細分析 app/routers/auth.py，重點檢查：
- 認證繞過（對照 foundry/rules/corpus/missing-auth-check.yaml）
- 硬編碼密碼（對照 foundry/rules/corpus/hardcoded-credentials.yaml）
- Timing side channel（對照 foundry/rules/corpus/timing-side-channel.yaml）

對每個問題，提供完整的三段式論證。

追蹤單一資料流

請追蹤 app/routers/orders.py 中 create_order() 函式裡
user_id 參數從 HTTP 請求進來到資料庫查詢的完整路徑，
確認是否有 IDOR 漏洞（參考 foundry/rules/corpus/idor.yaml）。

特別確認：有沒有驗證這個 user_id 屬於目前登入的使用者？

程式碼解析器

foundry/index/parser.py 使用 tree-sitter 解析 Python 程式碼， 提取函式層級的索引。可以獨立使用，不需要 LLM 或 API key。

from foundry.index.parser import index_file, file_hash

# 解析一個 Python 檔案，取得所有函式
records = index_file("app/routers/auth.py")

for rec in records:
    print(f"函式：{rec.symbol_name}")
    print(f"位置：第 {rec.start_line}–{rec.end_line} 行")
    print(f"內容：{rec.body[:100]}...")
    print()

依賴：

pip install tree-sitter tree-sitter-python

新增自訂規則

在 foundry/rules/corpus/ 新增一個 YAML 檔案：

# foundry/rules/corpus/my-rule.yaml
id: MY-001
name: My Custom Vulnerability
vulnerability_class: My Vuln
cwe: CWE-XX
severity_hint: high
applicable_languages: [python]
scope: function
prompt_template: |
  分析以下函式是否有 [漏洞描述]。
  需要同時滿足：
  1. [條件 A — 輸入來源]
  2. [條件 B — 缺少防護]
  3. [條件 C — 有安全影響]

  {{function_body}}

  只在三個條件都成立時回報漏洞。

新增後告訴 Claude Code：

請用 foundry/rules/corpus/my-rule.yaml 的條件，
掃描 app/ 目錄。

常見問題

和一般的 linter 或 SAST 工具有什麼不同？

一般工具找的是「有這個 pattern」，Foundry 的規則要求三段論證： 資料必須來自攻擊者可控的輸入、必須跨越信任邊界、必須有具體的安全影響。 誤報率低很多，每個回報都是真正可利用的問題。

Claude Code 可以掃多大的 codebase？

視 context window 大小而定。對於大型 repo，建議分目錄、分函式進行， 而不是一次貼所有程式碼。可以先用 foundry/index/parser.py 取得函式清單，再逐一分析高風險函式。

如何確保回報的問題不是誤報？

要求 Claude Code 提供完整的三段論證（reachability、trust boundary、impact）。 如果它無法提供具體的攻擊路徑，就要求它標記為「可疑」而非「確認漏洞」。 缺任何一段論證就不應接受為真正的漏洞。

如果之後想跑完整的自動化版本怎麼做？

需要額外取得 Anthropic API key（console.anthropic.com 有免費額度）， 設定 MySQL（Tailscale 連到 mysql.shiragaserver.lan）， 設定 GitHub token，然後參考 specs/001-foundry/quickstart.md。
